# Hailfreq Server Kit v0.3.0 — Turnkey Single-File Deployment Implementation Plan (Plan 10)

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Replace the current multi-file template-rendering kit with a single `compose.yml` that bundles all service configs inline + auto-generates secrets on first run. Operator's full deploy becomes: download one compose.yml, create a 4-line `.env`, run `docker compose up -d`. No `git clone`, no `setup.sh`, no template files.

**Architecture:** All service configs (Caddyfile, livekit.yaml, turnserver.conf, log.config, homeserver.yaml) move into top-level `configs:` blocks in compose.yml with `content: |` inline. Compose v2 substitutes `${VAR}` from `.env` at parse time, so the configs are operator-specific without per-deployment files. A short-lived `bootstrap` Alpine container runs on first start, generates 7 cryptographic secrets via `openssl rand`, writes them to a `hailfreq-secrets` Docker volume. Postgres consumes its secret via the standard `POSTGRES_PASSWORD_FILE`. Synapse uses a custom entrypoint script (also inline) that reads secrets from `/run/secrets/`, renders the final homeserver.yaml, then exec's `/start.py`. livekit-auth and coturn similarly source secrets from the shared volume. The result is a single compose.yml that the operator never needs to edit — only `.env`.

**Tech Stack:** Docker Compose v2 (or podman-compose with the equivalent feature set). Inline configs with `content: |`. Docker secrets via shared volume + `*_FILE` env conventions. No new code in any service container — just config and entrypoint wiring.

**Out of scope:**
- Migration of the existing rpk.chat deployment to v0.3 (stays on v0.2; v0.3 is fresh-deploy only)
- macOS support (out of scope per project, doesn't affect deployment kit)
- A web UI for first-run config (operator still edits .env in their shell)
- Auto-update of the running stack on new image release (operators run `docker compose pull && up -d` manually)

**Backwards compatibility:**
- Existing v0.2 deployments continue to run as-is (the v0.2 compose.yml + templates still exist in the v0.2.0 git tag)
- v0.3 deploys are NEW installs. No automatic migration path.
- Documented: operators with running v0.2 stacks should stay on v0.2 unless they want to redeploy from scratch.

**Repo location:** `/home/shreen/code/tactical-radio`. Commits go to `master`. SSH alias `hailfreq-vps` available (running v0.2 stack — do NOT touch for v0.3 testing; use a separate environment).

## Phase B — Bootstrap secrets container

### Task 4: Add bootstrap service that generates 7 cryptographic secrets

**Files:**
- Modify: `server/compose.yml`

The bootstrap container runs once on first start, generates secrets to a shared `hailfreq-secrets` volume. Subsequent runs detect the secrets already exist and exit cleanly.

- [ ] **Step 1: Add `bootstrap` service and `hailfreq-secrets` volume**

In `compose.yml`, at the top of the `services:` section:

```yaml
  bootstrap:
    image: docker.io/alpine:3.22
    container_name: hailfreq-bootstrap
    restart: "no"
    volumes:
      - hailfreq-secrets:/secrets
    command:
      - sh
      - -c
      - |
        set -e
        if [ -f /secrets/.bootstrap-complete ]; then
          echo "→ Secrets already generated, skipping bootstrap"
          exit 0
        fi
        echo "→ Generating secrets..."
        apk add --quiet --no-cache openssl
        for s in postgres_password synapse_registration_shared_secret synapse_macaroon_secret synapse_form_secret livekit_api_key livekit_api_secret turn_shared_secret; do
          openssl rand -hex 32 > /secrets/$$s
          chmod 600 /secrets/$$s
        done
        touch /secrets/.bootstrap-complete
        echo "→ Generated 7 secrets in /secrets"
    networks:
      - hailfreq
```

And add the volume to the `volumes:` top-level section:

```yaml
volumes:
  postgres_data:
  synapse_data:
  caddy_data:
  caddy_config:
  hailfreq-secrets:
```

(`$$s` is needed because compose treats `$` specially; doubled `$$` becomes literal `$s` in the shell.)

- [ ] **Step 1: Update livekit-auth's source to read secrets from files**

Add at the top of the file:

```ts
import fs from "node:fs";

function readSecret(name: string, envFallback: string): string {
  const filePath = `/run/secrets/${name}`;
  if (fs.existsSync(filePath)) {
    return fs.readFileSync(filePath, "utf-8").trim();
  }
  const env = process.env[envFallback];
  if (env) return env;
  throw new Error(`Secret ${name} not available — checked ${filePath} and env ${envFallback}`);
}

const LIVEKIT_API_KEY = readSecret("livekit_api_key", "LIVEKIT_API_KEY");
const LIVEKIT_API_SECRET = readSecret("livekit_api_secret", "LIVEKIT_API_SECRET");
```

Replace references to `process.env.LIVEKIT_API_KEY` with `LIVEKIT_API_KEY` (the constant). Same for API_SECRET.

### Task 10: Add Docker Hub push to the publish-livekit-auth workflow

**Files:**

Adds a parallel push to Docker Hub so operators can also pull from `docker.io/hailfreq/lk-auth`. Note the name is `lk-auth` (not `livekit-auth`) per the earlier discussion — short, matches the URL path.

**Pre-requisites the operator must complete first:**

2. Create `hailfreq/lk-auth` public repository on Docker Hub

- [ ] **Step 1: Update the workflow to push to both registries**

```yaml
name: Publish livekit-auth image

on:
  push:
    branches: [master]
    paths:
      - 'server/livekit-auth/**'
      - '.github/workflows/publish-livekit-auth.yml'
    tags:
      - 'v*'
  workflow_dispatch:

env:
  GHCR_IMAGE: ghcr.io/${{ github.repository_owner }}/livekit-auth
  DOCKERHUB_IMAGE: hailfreq/lk-auth

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write
    steps:
      - name: Checkout
        uses: actions/checkout@v4

      - name: Log in to GHCR
        uses: docker/login-action@v3
        with:
          registry: ghcr.io
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Log in to Docker Hub
        if: ${{ env.DOCKERHUB_USERNAME != '' }}
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}

      - name: Extract image metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: |
            ${{ env.GHCR_IMAGE }}
            ${{ env.DOCKERHUB_IMAGE }}
          tags: |
            type=raw,value=latest,enable=${{ github.ref == 'refs/heads/master' }}
            type=ref,event=tag
            type=semver,pattern={{version}}
            type=semver,pattern={{major}}.{{minor}}
            type=sha,format=short

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: ./server/livekit-auth
          file: ./server/livekit-auth/Dockerfile
          push: true
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max
```

The `if: ${{ env.DOCKERHUB_USERNAME != '' }}` guard on the Docker Hub login step skips Docker Hub push when the secrets aren't configured — useful for forks.

Wait, `env.DOCKERHUB_USERNAME` in this context refers to the workflow env, not the secret. To conditionally skip when secrets aren't set, use `if: ${{ secrets.DOCKERHUB_USERNAME != '' }}` instead. But secrets in `if:` need to go through an env var first. Cleaner: just always try, and let the step fail gracefully if secrets aren't set. Or use the documented pattern: pass secrets through a job-level env and check.

Simplest approach: don't make it conditional in the workflow. If secrets aren't configured, the login step fails and the workflow fails. For our case (rocked secrets), that's correct behavior.

```yaml
      - name: Log in to Docker Hub
        uses: docker/login-action@v3
        with:
          username: ${{ secrets.DOCKERHUB_USERNAME }}
          password: ${{ secrets.DOCKERHUB_TOKEN }}
```

(No conditional.)
